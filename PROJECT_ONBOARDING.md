# Project Onboarding Brief

## Purpose

This project is a Node.js converter for Adobe InDesign IDML files. It reads `.idml`
packages, extracts document structure and resources from the XML files inside them,
and emits JSON that is shaped for a Fabric.js-style canvas renderer.

The practical goal is not just "IDML to generic JSON"; the current code is moving
toward "IDML to renderable Fabric.js document JSON" with better visual fidelity for
geometry, transforms, paint, text, images, fonts, and page metadata.

## Current Status

The project is functional as a batch converter for the sample files in `input/`.
Running `npm start` processes all `.idml` files in `input/` and writes one `.json`
file per input into `output/`.

There is no formal test runner configured in `package.json`. `test.js` is a manual
benchmark/fixture script that processes all inputs and prints timing, page counts,
object counts, memory usage, and per-file failures.

Git status may fail on this machine with a "dubious ownership" warning unless the
repository is added to Git's safe directory list. That is an environment issue, not
a project code issue.

## Main Entry Points

- `index.js`: CLI-style batch entry point used by `npm start`.
- `src/idmlProcessor.js`: Core converter. This is where IDML archives are opened,
  XML files are parsed, pages/spreads are traversed, page items are converted, and
  final JSON is assembled.
- `src/colorOps.js`: Color, tint, spot color, gradient, RGB, and CMYK helpers.
- `src/imageOps.js`: Linked/embedded image extraction and base64/JPEG conversion.
- `src/characterStyles.js`: Character style extraction and `BasedOn` inheritance.
- `src/paragraphStyles.js`: Paragraph style extraction and `BasedOn` inheritance.
- `src/objectStyles.js`: Object style extraction and `BasedOn` inheritance.
- `src/utils.js`: Point-to-pixel conversion and affine matrix helpers.
- `test.js`: Manual converter benchmark across all files in `input/`.
- `todo.md`: Current feature parity checklist. Treat this as the best roadmap
  signal in the repo.
- `pp.json`: Appears to be a reference or target JSON shape from another system.
  Do not assume it is generated by the current converter without validating it.

## How The Converter Works

1. `processIdml(filePath)` reads the IDML file as a ZIP archive with `jszip`.
2. It parses key XML files with `xmldom`:
   - `designmap.xml`
   - `Resources/Styles.xml`
   - `Resources/Graphic.xml`
   - `Resources/Preferences.xml`
   - individual spread files from `Spreads/...`
   - individual story files from `Stories/...`
3. It extracts document-level data such as page size, bleed, margins, and origin.
4. It loops through spreads and pages, then processes page items into Fabric-like
   objects.
5. It resolves colors, gradients, styles, transforms, text runs, images, fonts, and
   diagnostics.
6. It returns final JSON with:
   - `designId`
   - `title`
   - `colorConversionPolicy`
   - `fonts`
   - `assets`
   - `pages`
   - `documentData`
   - `diagnostics`
   - `isvx`

## What It Currently Does

- Converts IDML page items into Fabric.js-style canvas objects.
- Handles multiple pages and spreads.
- Normalizes object positions relative to each page rather than spread origin.
- Converts point units to pixels with a fixed conversion factor in `pt_px`.
- Processes these IDML item types:
  - `Rectangle`
  - `TextFrame`
  - `Polygon`
  - `GraphicLine`
  - `Oval`
  - `SplineItem`
  - `Image`
  - `EPS`
  - `SVG`
  - `Group`
- Preserves object order with `zIndex`.
- Supports nested group traversal.
- Decomposes IDML affine transforms into Fabric-compatible fields:
  - `left`
  - `top`
  - `scaleX`
  - `scaleY`
  - `flipX`
  - `flipY`
  - `angle`
  - `skewX`
  - `skewY`
- Resolves object styles, paragraph styles, and character styles, including
  `BasedOn` inheritance.
- Applies inline style overrides where currently implemented.
- Extracts text from story XML and emits Fabric `Textbox` data.
- Emits per-character style differences in Fabric-style `styles` data.
- Converts CMYK/RGB color definitions to renderable RGB strings while preserving
  CMYK metadata.
- Handles tints, swatches, spot colors, linear gradients, radial gradients, opacity,
  blend modes, stroke caps, joins, dashes, stroke alignment metadata, and corner
  radius metadata.
- Adds diagnostics warnings for unsupported overprint behavior.
- Extracts linked and embedded images into base64 data URLs where possible.
- Extracts font metadata and embeds matching font files from `Document Fonts/`.

## What It Does Not Do Yet

The converter is not a complete IDML renderer and should not be treated as visually
lossless. Known gaps from `todo.md` and the codebase include:

- Threaded story flow across linked text frames.
- Full paragraph/character override parity.
- Kerning, tracking, leading, baseline shift, first baseline behavior, OpenType
  features, small caps, superscript/subscript, and vertical justification parity.
- Text-on-path fidelity beyond partial extraction.
- Drop shadows, inner shadows, feathering, glow, transparency effects, and exact
  effect render order.
- Object clipping and masks.
- Layer visibility, locking, printability, and layer-level semantics.
- Master page and override support.
- Full facing-page/spread coordinate normalization.
- Complete bleed, slug, and page-box mapping.
- Precise image frame/content transform separation.
- Crop and fitting option parity.
- Linked asset resolution outside files embedded in the IDML package.
- ICC/profile-aware color conversion.
- Tables, bullets, numbering, anchored objects, inline objects, footnotes,
  endnotes, hyperlinks, notes, conditions, and text wrap.
- Versioned output JSON schema.
- Robust diagnostics for every unsupported feature.
- Automated golden-file or visual regression tests.

## What It Should Become

The project should become a predictable conversion layer with a documented schema,
clear feature support boundaries, and regression coverage across representative
IDML fixtures.

The likely product direction is:

- Input: `.idml` exported from Adobe InDesign.
- Output: stable JSON that can be loaded into a Fabric.js or similar canvas editor.
- Quality bar: structural correctness first, then visual fidelity, then unsupported
  feature diagnostics.
- Operating model: deterministic batch conversion with no Adobe InDesign runtime
  dependency.

## Important Implementation Notes

- IDML files are ZIP archives. Most useful data is in XML files under `Resources/`,
  `Spreads/`, and `Stories/`.
- The converter assumes `Resources/Styles.xml`, `Resources/Graphic.xml`, and
  `Resources/Preferences.xml` exist. Missing-resource handling is limited.
- Style caches are cleared at the start of every `processIdml` call to avoid style
  leakage between files.
- `pt_px` currently uses `1.333` as the point-to-pixel conversion factor. Any change
  here affects every geometry and typography output.
- Transform handling is centralized in `multiplyAffineMatrices` and
  `decomposeTransform`. Touch these carefully and validate against fixtures.
- Some generated output fields are Fabric-native, while others are IDML metadata
  retained for later rendering or diagnostics.
- The current `clipPath` dimensions include hard-coded additions to page width and
  height. This should be reviewed before treating output geometry as stable.
- `diagnostics.warnings` currently exists but is underused. Expanding it is probably
  better than silently dropping unsupported IDML constructs.
- `output/` contains generated fixture JSON and can be useful for manual comparison,
  but it is not backed by an automated approval test.

## Setup

```bash
npm install
```

`canvas` has native dependencies. On Windows, installation may require Visual Studio
Build Tools and a compatible Node.js version.

## Common Commands

```bash
npm start
```

Processes every `.idml` file in `input/` and writes JSON into `output/`.

```bash
node test.js
```

Runs the manual benchmark/fixture processor and prints conversion stats.

There are no configured `npm test`, `npm run lint`, or `npm run build` scripts at
the time this brief was written.

## Suggested First Read For A New Developer

1. Read `README.md` for the intended high-level behavior.
2. Read `todo.md` for the actual feature parity roadmap.
3. Read `src/idmlProcessor.js` from top to bottom. Most behavior is nested inside
   `processIdml`, so this file is the main mental model.
4. Read `src/utils.js` before changing any transform or coordinate behavior.
5. Read `src/colorOps.js` before changing paint, swatches, gradients, or CMYK/RGB
   handling.
6. Compare one small input/output pair, such as `input/bounded-text.idml` and
   `output/bounded-text.json`.

## Recommended Next Work

1. Add a real test script to `package.json`.
2. Build fixture-based structural tests for all files in `input/`.
3. Add a versioned output schema and document required/optional fields.
4. Expand diagnostics so unsupported IDML constructs are reported consistently.
5. Create a feature parity matrix from `todo.md` with statuses: supported, partial,
   unsupported, not evaluated.
6. Add golden JSON snapshots for stable fixtures.
7. Add visual regression testing if there is a downstream Fabric.js renderer.
8. Fix or justify hard-coded page/clip dimensions and zoom defaults.
9. Improve text fidelity before expanding into rare layout features.
10. Define how `pp.json` relates to converter output and whether it is a target,
    fixture, or legacy artifact.

## Risk Areas

- Text fidelity is the largest remaining area of uncertainty.
- Image placement likely needs careful work because InDesign frame transforms and
  content transforms are not the same thing.
- Facing pages and master pages can create subtle coordinate bugs.
- Color conversion is currently a deterministic device-CMYK policy, not
  ICC-accurate output.
- Unsupported effects can make output look visibly different even when structure is
  correct.
- The core processor is large and heavily nested, so small changes can have wide
  conversion impact.

## Good Takeover Strategy

Start by making the current behavior measurable before adding features. The best
first milestone is a regression harness that runs sample IDML files, validates
basic object/page/style counts, records diagnostics, and compares stable JSON
fields. After that, prioritize one fidelity area at a time, starting with text and
image placement, because those are most visible in real documents.
